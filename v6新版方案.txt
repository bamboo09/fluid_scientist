# 完整改造任务

## Fluid Scientist 开放世界需求理解与通用能力改造方案

## 一、任务目标

本轮必须解决以下根本问题：

1. 用户提出当前 Schema 中没有的新几何时，不能忽略；
2. 用户改变流体、材料属性、边界条件、初始条件、物理模型、观测指标时，不能依赖逐个新增模板；
3. 大模型必须完整保存用户显式表达，不得只把内容映射到现有枚举；
4. 已有能力可以继续使用确定性 Compiler；
5. 新需求能够被转换为通用表达时，不得生成新代码；
6. 当前确实无法表达或编译时，必须进入能力缺口流程；
7. 所有无法处理的内容必须展示给用户，绝不允许静默丢失；
8. Skill、Prompt、能力检查和未知能力扩展必须进入真实生产主链。

本轮严禁只增加：

```text
TrapezoidSpec
TrapezoidCompiler
trapezoid_template
_extract_trapezoid()
if shape == "trapezoid"
```

梯形案例必须主要通过**通用多边形能力**解决，而不是专门模板。

# 二、当前问题的准确判断

截图中的用户输入明确包含：

```text
上底2m
下底4m
高3m
梯形凸起
位于圆柱正下方
贴附下壁面
```

但是右侧正式参数中：

* 只有圆柱；
* 没有第二个几何实体；
* 没有梯形参数；
* 没有 unsupported capability；
* 没有提示“系统无法表达梯形”；
* 系统只针对上边界提出澄清。

这至少暴露五个问题。

## 2.1 Schema是封闭枚举，而不是开放实体列表

当前结构接近：

```python
class CylinderFlowSpec:
    cylinder: CylinderSpec
    rectangle: RectangleSpec
    triangle: TriangleSpec
    bottom_profile: BottomProfileSpec
```

这意味着只有预先定义好的槽位才能进入 Spec。

梯形没有槽位，就可能直接消失。

正确结构应该是：

```python
class ResearchSpec:
    geometry_entities: list[GeometryEntity]
```

新几何先进入实体列表，再判断如何表达和编译。

## 2.2 语义识别和能力支持被混成一件事

错误逻辑：

```text
系统不支持梯形
→ 所以不识别梯形
```

正确逻辑：

```text
用户明确提出梯形
→ 必须识别并保存梯形
→ 再检查系统是否支持编译
```

理解层必须忠实于用户，不能因为编译器不支持就篡改或删除用户需求。

## 2.3 缺少“输入覆盖率检查”

系统没有检查：

```text
用户显式提到的每一个实体、参数、边界、指标
是否都进入了正式Spec
```

因此即使丢掉整个梯形，流程仍然继续。

## 2.4 大模型被迫做模板分类

当前模型主要在判断：

```text
这是 cylinder？
rectangle？
triangle？
cosine_bell？
```

而不是先回答：

```text
用户描述了哪些物理实体？
这些实体具有什么几何参数和空间关系？
它们可以用何种通用数学表示？
```

## 2.5 参数工作台直接依赖固定Schema

后端丢掉梯形后，前端自然无法动态展示。

所以问题不只是前端，也不只是提示词，而是：

```text
意图理解
→ 中间表示
→ 能力检查
→ 动态Schema
```

整条链都需要调整。

# 三、目标架构

改造后采用以下链路：

```text
用户原文
  ↓
Mention Inventory：显式信息清点
  ↓
Open-World Intent Extractor：开放世界实体抽取
  ↓
Semantic Representation Planner：通用数学表达
  ↓
Semantic Critic：遗漏、替换、重复、冲突检查
  ↓
Clarification Planner：只询问真正阻塞项
  ↓
Canonical Research IR
  ↓
Dynamic Schema Builder：动态参数工作台
  ↓
Capability Planner
  ├─ 已有通用能力可完成
  │    ↓
  │  Generic Compiler
  │
  └─ 当前能力无法完成
       ↓
     Missing Capability
       ↓
     Config Extension / Compiler Extension
  ↓
用户确认
  ↓
Case生成和OpenFOAM验证
```

关键原则：

```text
语义表示不等于Compiler支持情况
用户需求不等于模板名称
新名词不一定需要新代码
未知内容不能被忽略
```

### 梯形不需要专用Compiler

梯形可表示为：

```json
{
  "type": "parametric_polygon",
  "subtype": "trapezoid",
  "definition": {
    "top_width": 2.0,
    "bottom_width": 4.0,
    "height": 3.0,
    "attachment": "bottom_wall",
    "horizontal_alignment": "centered_under:cylinder_1"
  }
}
```

在当前案例中，圆柱横坐标为 75m，可进一步确定顶点：

```text
(73, 0)
(77, 0)
(76, 3)
(74, 3)
```

然后交给通用 Polygon Compiler。

## 4.4 同一个编译能力覆盖多种形状

建立以下通用几何编译器：

```text
CircleGeometryCompiler
PolygonGeometryCompiler
ProfileFunctionCompiler
ImportedMeshCompiler
CSGGeometryCompiler
```

其中：

```text
triangle
rectangle
trapezoid
parallelogram
pentagon
user-provided polygon
```

全部交给：

```text
PolygonGeometryCompiler
```

而不是分别增加六个模板。

### 强制验收

梯形功能完成后，代码中不得新增：

```text

_compile_trapezoid_case
```

允许存在语义归一化或参数转顶点函数，但最终编译必须走通用 polygon 路径。

# 五、材料与流体也必须开放化

不要只解决几何。

新增：

```python
class MaterialIntent(BaseModel):
    material_id: str
    raw_name: str
    phase: Literal["gas", "liquid", "solid", "multiphase", "unknown"]

    model: Literal[
        "incompressible_newtonian",
        "compressible_newtonian",
        "non_newtonian",
        "multiphase",
        "custom",
        "unknown"
    ]

    properties: dict[str, ParameterValue]
    source_spans: list[str]

    missing_required_properties: list[str]
    capability_status: str
```

例如用户说：

```text
使用运动黏度为3e-5 m²/s、密度850 kg/m³的自定义油品
```

系统必须生成：

```json
{
  "raw_name": "自定义油品",
  "model": "incompressible_newtonian",
  "properties": {
    "density": 850,
    "kinematic_viscosity": 0.00003
  }
}
```

不能因为材料库里没有这种油，就退回空气或水。

材料解析优先级：

```text
用户明确属性
> 用户明确Re、U和特征长度推导
> 版本化材料数据库
> 模型建议并要求确认
> unknown_required
```

禁止：

```text
未识别流体 → 默认空气
```

# 六、边界条件必须使用语义模型

当前不能把边界理解成固定的左、右、上、下模板。

新增：

```python
class BoundaryIntent(BaseModel):
    boundary_id: str
    target: BoundarySelector
    physical_role: Literal[
        "velocity_inlet",
        "mass_flow_inlet",
        "pressure_inlet",
        "pressure_outlet",
        "open_boundary",
        "no_slip_wall",
        "slip_wall",
        "moving_wall",
        "symmetry",
        "periodic",
        "shear_stress",
        "heat_flux",
        "convective_outlet",
        "custom",
        "unknown"
    ]

    quantities: dict[str, ParameterValue]
    raw_text: str
    source_span: str

    semantic_status: str
    capability_status: str
```

然后再由 OpenFOAM v13 Boundary Lowering 负责：

```text
velocity_inlet
→ U fixedValue
→ p zeroGradient

pressure_outlet
→ p fixedValue
→ U zeroGradient
```

边界的物理语义和 OpenFOAM patchField 类型不能在意图层混为一谈。

遇到新边界时：

```text
先保存 physical_role
→ 检查 Foundation v13 是否有配置映射
→ 有则生成
→ 无则进入MissingCapability
```

不能忽略，也不能直接改成最相近的边界。

# 十二、Prompt 2：几何通用表示规划

文件：

```text
prompts/geometry_representation_planner.md
```

内容：

```text
你是 CFD 几何语义与数学表示规划器。

输入包括：
1. 用户原文；
2. 已抽取的几何实体；
3. 已抽取的空间关系；
4. 当前通用几何表示能力。

你的任务不是选择具体OpenFOAM模板，而是为每个几何实体选择忠实、最通用的数学表示。

可用表示：

- circle
- ellipse
- parametric_polygon
- explicit_polygon
- profile_function
- constructive_solid_geometry
- imported_mesh
- implicit_surface
- unknown

【核心规则】

1. 优先使用已有通用表示，而不是创建具体形状模板。
2. triangle、rectangle、trapezoid、parallelogram、regular_polygon等优先表示为polygon。
3. 正弦、余弦、分段函数型壁面优先表示为profile_function。
4. 用户提供顶点时使用explicit_polygon。
5. 用户提供CAD/STL时使用imported_mesh。
6. 无法由现有参数唯一确定时，标记needs_clarification。
7. 不得为了可编译而修改用户几何。
8. 不得丢弃实体。
9. 不得把未知形状转换为最相近形状。
10. 仅当数学含义确定时，才能计算顶点。

【梯形示例】

输入：
上底2m、下底4m、高3m的梯形凸起，贴附下壁面，位于圆柱正下方。
```

# 十九、未知能力的处理层级

新情况出现时，必须按以下优先级处理。

## Level 1：现有通用表示

例如：

```text

五边形 → polygon
任意给定顶点 → polygon
正弦凸起 → profile_function
```

不生成代码。

## Level 2：配置组合

例如：

```text
已有OpenFOAM边界条件的新参数组合
已有functionObject的新采样位置
```

只生成配置。

## Level 3：通用组件扩展

例如当前没有 ellipse compiler，但可以增加：

```text
EllipseGeometryCompiler
```

此时新增的是一类数学表示，不是每个实验一个模板。

## Level 4：外部几何输入

非常复杂形状要求用户上传：

```text
STL
OBJ
polyMesh
CAD转换文件
```

## Level 5：代码扩展

只有前四级无法完成时进入 ExtensionOrchestrator。

# 二十七、完成标准

只有同时满足以下条件才能结束：

1. 截图中的梯形不再消失；

5. 梯形、三角形、矩形和五边形共用通用Polygon Compiler；
6. 更换流体时不需要增加场景模板；
7. 更换边界时先进行语义表达和能力检查；
8. 更换指标时能动态生成MeasurementPlan；

13. 真实GLM和真实OpenFOAM E2E通过；
14. 不存在按测试句子硬编码；
15. 输出完整证据报告。

最终目标不是“支持更多固定案例”，而是：

> **用户提出新的形状、材料、边界、初始条件、物理模型或指标时，系统都能先忠实理解并保存，再通过通用表示、能力组合或受控扩展解决；即使暂时无法完成，也绝不忽略和伪装。**
