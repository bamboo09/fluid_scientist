# Fluid Scientist 新分支完整重构方案 v5

## 复杂自然语言研究计划 → 只读实验草案 → 多轮确认 → CasePlan → OpenFOAM 能力检查与扩展闭环

---

# 0. 一句话目标

本分支不是继续做“参数表工作台”，也不是做普通聊天机器人。

本分支要把 Fluid Scientist 改造成：

```text
Session 级科研实验草案智能体
+ 复杂自然语言研究计划分解
+ 只读实验草案卡片
+ 多轮自然语言修改
+ 用户确认后才更新版本
+ confirmed 后生成 CasePlan
+ CasePlan 能力检查
+ 缺能力进入 CodeExtension 闭环
```

最终用户体验应该是：

```text
用户可以直接输入复杂 CFD 研究描述；
系统先拆解研究任务，而不是乱套模板；
系统明确识别几何、物理模型、边界条件、初始条件、观测指标、缺失信息；
系统生成只读实验草案；
用户通过自然语言修改草案；
每次修改先生成 ChangeProposal；
用户确认后才真正更新草案；
草案 confirmed 后才允许生成 OpenFOAM Case；
当前能力不足时，系统明确说明缺什么，而不是假装生成 Case。
```

---

# 1. 当前必须解决的核心问题

当前方案最大问题不是缺少某个类，而是主链路不够清楚。

必须解决以下问题：

```text
1. 用户一次输入多个复杂研究计划时，系统不能把它们混成一个 case。
2. 用户输入很长、很科研化时，系统不能只提取几个参数，然后生成空表。
3. 用户描述中包含“观测机理目标”时，系统不能误当成 OpenFOAM 参数。
4. 用户描述不完整时，系统不能直接失败，也不能瞎补高风险参数。
5. 用户提出新参数、新边界条件、新指标时，系统不能说模板没有。
6. LLM 不能直接修改真实 draft，只能生成 proposal。
7. 用户确认前，任何修改都不能生效。
8. confirmed 后的版本必须冻结，不能原地修改。
9. 生成 Case 前必须先生成 CasePlan，再做能力检查。
10. 缺能力时必须进入 MissingCapability / CodeExtension，而不是生成假 Case。
```

# 2. 总体架构

完整链路如下：

```text
用户自然语言输入
→ InputRouter
→ NaturalLanguageStudyDecomposer
    → StudySplitter
    → StudyTypeClassifier
    → PhysicsFrameExtractor
    → ParameterExtractor
    → ConditionExtractor
    → ObservableExtractor
    → AmbiguityDetector
    → CapabilityPreChecker
→ BatchStudyPlan / StudyIntent
→ 用户选择一个 study 或生成多个草案
→ DraftSession
→ ResearchState
→ ClarificationPlanner
→ ExperimentDraft 只读草案
→ Validator
→ 用户自然语言修改
→ ChangeProposal
→ 用户确认 / 取消
→ ApplyProposalExecutor
→ draft version + 1
→ ready
→ 用户确认实验版本
→ confirmed / locked
→ CasePlanner
→ CapabilityRegistry
    → 支持：NativeCompiler 生成 OpenFOAM Case
    → 不支持：MissingCapability
        → CodeExtensionSpec
        → LLM code_generation
        → sandbox test
        → approve
        → register capability
        → 回到 CasePlanner
→ Run
→ Ingest
→ Analyze
→ ScientificReport
```

# 3. 产品形态

## 3.1 不做典型 chatbot

页面不要做成纯聊天窗口。

推荐结构：

```text
顶部：研究任务 / 实验草案标题
中部：只读实验草案卡片
右侧：状态、缺失信息、能力检查、LLM 调用记录
底部：自然语言修改输入框 + 事件流
```

用户不直接编辑参数表，而是输入：

```text
把入口速度改成 0.2m/s。
增加入口湍流强度 5%。
输出出口速度均匀性。
壁面热通量作为输出，不作为边界条件。
这个 case 改成 LES。
```

系统返回 proposal，用户确认后才更新。

---

## 3.2 只读实验草案卡片

草案卡片至少包含：

```text
1. 研究目标
2. 研究类型
3. 几何与计算域
4. 流体 / 材料
5. 控制参数
6. 初始条件
7. 边界条件
8. 物理模型
9. 湍流 / 热 / 浮力 / 运动模型
10. 网格策略
11. 求解器建议
12. 数值设置
13. 观测指标
14. 后处理计划
15. 已知条件
16. 系统假设
17. 高风险缺失
18. 当前能力检查结果
19. 是否可生成 Case
```

每个参数必须有来源：

```text
user_provided
derived
system_recommended
assumption
unknown_required
capability_default
```

# 4. 第一层：复杂自然语言研究计划分解

## 4.1 为什么要有 StudyDecomposer

用户可能输入：

```text
1. 近壁倾斜圆柱 Re=3900 三维湍流尾迹……
2. 倾斜圆射流 45 度冲击平壁……
3. 倾斜椭圆柱 Re=5000 非定常绕流……
4. 密度分层流体中水平圆柱垂向振荡……
5. 后台阶流动 Re=5000 分离再附……
```

系统第一步不能进入 draft，也不能直接 compile。

必须先判断：

```text
这是一个 batch_research_request；
其中包含 5 个独立研究任务；
每个任务需要独立生成 StudyIntent；
每个任务独立进入草案、澄清、能力检查、CasePlan。
```

---

## 4.2 新增 NaturalLanguageStudyDecomposer

新增模块：

```python
class NaturalLanguageStudyDecomposer:
    def decompose(self, user_message: str) -> BatchStudyPlan:
        ...
```

内部包含：

```text
1. StudySplitter：识别单任务还是多任务；
2. StudyTypeClassifier：识别 CFD 研究类型；
3. PhysicsFrameExtractor：提取物理框架；
4. ParameterExtractor：提取 Re、Fr、角度、间距、长短轴比等；
5. ConditionExtractor：提取初始条件和边界条件；
6. ObservableExtractor：提取观测指标和机理目标；
7. AmbiguityDetector：识别缺失、冲突和歧义；
8. CapabilityPreChecker：初步判断当前系统能否落地；
9. PriorityRanker：推荐优先落地任务。
```

---

## 4.3 BatchStudyPlan

```python
class BatchStudyPlan(BaseModel):
    batch_id: str
    input_type: Literal["single_study", "batch_study"]

        "select_one_to_continue",
        "generate_all_draft_summaries",
```

对于用户一次输入 5 个研究计划，系统应返回：

```text
我识别到 5 个独立 CFD 研究任务。
建议先生成 5 个草案摘要，然后选择一个进入详细草案和 CasePlan。
```

# 5. 复杂 CFD 研究描述的拆解规则

每个 StudyIntent 必须拆成 8 类信息。

## 5.1 研究对象

提取：

```text
物理对象
几何类型
是否近壁
是否倾斜
是否运动
是否三维
是否涉及热
是否涉及浮力
是否涉及湍流
```

例如：

```text
near_wall_inclined_cylinder_wake
inclined_round_impinging_jet
inclined_elliptic_cylinder_near_wall_wake
stratified_oscillating_cylinder_wake
backward_facing_step_separation
```

---

## 5.2 控制参数

提取：

```text
Re
Fr
D
H
倾角
间距
间隙比
长短轴比
扩张比
入口边界层厚度
振荡幅值
振荡频率
```

重要规则：

```text
Re 和 Fr 不是 OpenFOAM 可直接输入。
必须继续推导或追问：
- 特征长度是什么？
- 参考速度是什么？
- 运动粘度是多少？
- 流体密度是多少？
- Fr 的定义是什么？
```

---

## 5.3 初始条件

识别：

```text
初始全场静止
初始线性密度分层
初始充分发展湍流边界层
随后施加均匀来流
随后施加充分发展管流入口
```

标准化为：

```text
initial_velocity_field
initial_pressure_field
initial_density_field
initial_temperature_field
initial_turbulence_field
startup_strategy
```

注意：

```text
“初始全场静止，随后施加来流”
不是一句普通描述。
它意味着 transient startup，需要定义 t=0 后入口如何施加。
```

---

## 5.4 边界条件

识别并标准化：

```text
入口给定速度剖面 → inlet.velocity_profile
圆柱无滑移 → wall.no_slip
壁面无滑移 → wall.no_slip
出口对流边界 → outlet.advective_or_equivalent
出口压力出口 → outlet.pressure
展向周期性 → spanwise.periodic
上下边界自由滑移 → top_bottom.free_slip
```

注意：

```text
“对流边界”不是一个直接可写入 OpenFOAM 的统一名称。
必须映射到具体 boundary condition，例如 advective、inletOutlet、zeroGradient、waveTransmissive 等。
映射前需要 Validator 检查。
```

---

## 5.5 物理模型

识别：

```text
3D
transient
turbulent
thermal
stratified density
buoyancy
moving body
periodic spanwise
wall-bounded flow
```

标准化为：

```text
dimension
temporal_type
turbulence_model_candidate
thermal_model
buoyancy_model
density_model
moving_body_model
solver_family_candidate
```

重要规则：

```text
用户说“湍流”不等于系统知道用 LES、RANS、DES 还是 DNS。
用户说“热通量”不等于一定要解能量方程。
用户说“密度分层”不等于一定能直接用当前 compiler 支持。
用户说“垂向振荡”必须进一步确认运动边界实现方式。
```

---

## 5.6 观测指标

识别：

```text
尾迹偏斜
展向翻转
壁面涡结构
马蹄涡
非对称分离
阻力系数
升力系数
频谱特性
壁面压力
热通量
再附点
回流区长度
雷诺应力
斯特劳哈尔数
混合层厚度
内波辐射
```

这些进入：

```text
MeasurementPlan
PostprocessPlan
ScientificAnalysisGoal
```

不要当成基础 case 参数。

---

## 5.7 机理目标

识别：

```text
揭示几何不对称与壁面约束耦合机制
揭示倾斜角诱导的复杂分离与转捩机理
揭示浮力-惯性耦合机制
揭示大尺度结构对分离泡稳定性的控制机制
```

这些进入：

```text
analysis_goals
```

不能进入 OpenFOAM 字典。

---

## 5.8 缺失信息

缺失信息分三级：

```text
Level 0：不影响理解，可默认推荐
Level 1：影响质量，需要确认，但不阻塞草案
Level 2：阻塞 Case 生成，必须澄清
```

例子：

```text
计算域尺寸未知 → needs_confirmation
网格分辨率未知 → needs_confirmation
湍流模型未知 → needs_confirmation 或 blocking，视 case 而定
几何尺寸完全未知 → blocking_for_case_generation
热通量是输入还是输出未知 → blocking_for_case_generation
振荡幅值和频率未知 → blocking_for_case_generation
密度分层公式未知 → blocking_for_case_generation
```

# 6. 对 5 个复杂输入的预期拆解

## 6.1 Case 1：近壁倾斜圆柱 Re=3900 三维湍流尾迹

识别：

```text
study_type = near_wall_inclined_cylinder_wake
dimension = 3D
temporal = transient
flow_regime = turbulent / transitional wake
Re = 3900
wall_effect = true
spanwise_boundary = periodic
```

已知条件：

```text
初始全场静止
随后施加均匀来流
入口给定速度剖面
圆柱与壁面无滑移
出口对流边界
展向周期
观测尾迹偏斜、展向翻转、壁面涡、阻力、升力、频谱
```

缺失：

```text
圆柱直径 D
倾斜角
圆柱与壁面间隙
计算域尺寸
展向长度
湍流模型
流体物性
时间步长和总时间
对流出口 OpenFOAM 映射
```

前 3 个澄清问题：

```text
1. 请确认圆柱直径 D、倾斜角和圆柱到壁面的间隙。
2. 湍流模拟采用 LES、RANS、DES，还是先使用系统推荐方案？
3. 是否接受系统按 D 推荐计算域尺寸、展向长度和初始网格分辨率？
```

能力预判：

```text
可能缺 inclined cylinder near-wall mesh generator。
需要三维周期边界支持。
需要 forceCoeffs 参考量。
需要频谱后处理。
```

---

## 6.2 Case 2：45° 倾斜圆射流冲击平壁 Re=23000

识别：

```text
study_type = inclined_round_impinging_jet
dimension = 3D
temporal = transient
flow_regime = turbulent jet
Re = 23000
inclination_angle = 45deg
nozzle_wall_distance = 2D
```

已知条件：

```text
初始全场静止
充分发展管流入口
壁面无滑移
出口压力出口
展向周期
观测壁面压力、热通量不对称、马蹄涡、非对称分离
```

关键歧义：

```text
“壁面热通量”到底是：
A. 输出结果；
B. 给定热边界条件；
C. 需要计算换热系数 / Nusselt 数；
D. 需要解能量方程。
```

缺失：

```text
D 的具体值
流体物性
入口充分发展管流生成方式
是否求解温度场
喷嘴长度
计算域尺寸
湍流模型
```

前 3 个澄清问题：

```text
1. 壁面热通量是作为输出结果分析，还是作为给定边界条件？
2. 入口充分发展管流希望用解析剖面、precursor，还是 mapped inlet？
3. 是否接受系统按 D 推荐计算域、喷嘴长度和网格？
```

能力预判：

```text
可能缺 impinging jet geometry generator。
可能缺 thermal solver 配置。
可能缺 wall heat flux postprocess。
```

## 6.3 Case 3：倾斜椭圆柱 Re=5000 非定常绕流

识别：

```text
study_type = inclined_elliptic_cylinder_near_wall_wake
dimension = 3D
temporal = transient
Re = 5000
aspect_ratio = 2.0
inclination_angle = 30deg
gap = 0.3 * minor_axis
```

已知条件：

```text
初始全场静止
随后施加均匀来流
入口抛物型速度剖面
椭圆柱表面无滑移
底部壁面无滑移
出口对流
展向周期
观测不对称涡脱落、壁面二次涡、展向翻转、升阻力和频谱
```

缺失：

```text
短轴长度 b 或长轴长度 a
Re 的特征长度定义
入口抛物型速度剖面对应通道高度
上边界条件
计算域尺寸
展向长度
湍流模型
```

前 3 个澄清问题：

```text
1. Re=5000 的特征长度采用短轴、长轴还是等效直径？
2. 入口抛物型速度剖面对应的通道高度是多少？
3. 上边界采用自由滑移、远场还是无滑移壁面？
```

能力预判：

```text
可能缺 elliptic cylinder near-wall mesh generator。
需要 forceCoeffs 支持非圆柱参考面积。
需要频谱后处理。
```

## 6.4 Case 4：密度分层流体中垂向振荡水平圆柱

识别：

```text
study_type = stratified_oscillating_cylinder_wake
dimension = 3D likely
temporal = transient
Re = 500
Fr = 0.2
physics = density stratification + buoyancy + moving body
```

已知条件：

```text
初始全场静止
线性密度分层
随后施加均匀来流
圆柱无滑移
上下边界自由滑移
展向周期
观测内波辐射、剪切层失稳、分层尾迹、St 偏移、混合层厚度
```

缺失：

```text
圆柱直径 D
振荡幅值
振荡频率
Fr 定义
Brunt–Väisälä 频率 N
密度分层函数 rho(z)
是否采用 Boussinesq 近似
重力方向
运动边界实现方式
```

前 3 个澄清问题：

```text
1. 垂向振荡的幅值和频率是多少？
2. Fr=0.2 是否定义为 U/(N·D)？
3. 是否采用 Boussinesq 近似和线性密度分层 rho(z)=rho0+alpha*z？
```

能力预判：

```text
大概率 not_compilable_yet。
可能缺 buoyancy model writer。
可能缺 density stratification initializer。
可能缺 oscillating cylinder dynamic mesh / overset 支持。
可能缺 internal wave 和 mixing layer 后处理。
```

## 6.5 Case 5：后台阶 Re=5000 分离与再附

识别：

```text
study_type = backward_facing_step_separation_reattachment
dimension = 3D
temporal = transient
Re = 5000
expansion_ratio = 1.5
inlet_boundary_layer_thickness = 0.3H
```

已知条件：

```text
台阶高度 H
入口 1/7 幂律速度剖面
上下壁面无滑移
出口对流
展向周期
观测回流区长度、再附点、剪切层涡配对、壁面压力恢复、雷诺应力
```

缺失：

```text
H 的具体值
Re 的定义：基于 H 还是入口水力直径
上游入口段长度
下游出口段长度
展向宽度
湍流模型
初始充分发展湍流边界层与入口 1/7 幂律之间的关系
```

前 3 个澄清问题：

```text
1. Re=5000 是基于台阶高度 H、入口通道高度，还是水力直径？
2. 入口 1/7 幂律是入口速度边界，还是已有充分发展初始场？
3. 是否接受系统按 H 推荐上游、下游和展向尺寸？
```

能力预判：

```text
推荐优先级最高。
原因：几何和边界条件最接近当前可落地 OpenFOAM case。
```

---

# 7. 推荐优先级策略

对于复杂 batch 输入，系统应自动给出落地优先级：

```text
优先级 1：后台阶流动
原因：几何标准，边界条件清楚，OpenFOAM 可落地性最高。

优先级 2：倾斜椭圆柱近壁绕流
原因：物理较清楚，但需要椭圆柱近壁网格能力。

优先级 3：近壁倾斜圆柱三维湍流尾迹
原因：三维、近壁、倾斜、湍流、周期边界，网格和湍流设置复杂。

优先级 4：倾斜圆射流冲击平壁
原因：入口充分发展管流、热通量/换热、冲击射流更复杂。

优先级 5：分层流体中振荡圆柱
原因：密度分层、浮力、内波、运动边界，当前系统大概率需要能力扩展。
```

这不是拒绝复杂 case，而是让用户知道：

```text
哪个能先跑通；
哪个需要澄清；
哪个需要扩展编译能力；
哪个暂时只能生成研究草案和 MissingCapability。
```

# 15. Confirmed 版本冻结

## 15.1 ready → confirmed

当 blocking issues 为 0，Validator 通过，状态进入 ready。

用户点击：

```text
确认实验版本
```

系统执行：

```text
draft.status = confirmed
draft.locked = true
session.status = confirmed
confirmed_version = current_version
```

---

## 15.2 confirmed 后不能原地修改

用户在 confirmed 后说：

```text
把入口速度改成 0.3m/s。
```

系统不能直接改当前版本。

正确流程：

```text
clone confirmed draft v1 → draft v2
v2.status = draft
v2.locked = false
在 v2 上生成 ChangeProposal
用户确认后更新 v2
v1 保持不变
```

```text

→ CodeExtensionSpec
→ LLM code_generation
→ 生成代码草案
→ 静态检查
→ 单元测试
→ sandbox run
→ LLM code_review
→ 人工确认

→ 回到 CasePlanner
→ 重新尝试 compile
```


# 24. 给 Codex / Trae 的直接指令

这个分支的优先级不是继续扩展更多复杂物理场景。

第一优先级是：

```text
把复杂自然语言研究计划拆对。
```

第二优先级是：

```text
把只读草案、多轮修改、proposal 确认、版本冻结、CasePlan 能力检查跑通。
```

第三优先级才是：

```text
扩展更多 OpenFOAM 编译能力。
```

禁止继续出现：

```text
1. 把多个研究任务合并成一个 case。
2. 参数全空。
3. 无关参数混入。
4. 用户回答澄清问题被当成新任务。
5. 用户修改参数后直接写入 draft。
6. proposal 未确认就生效。
7. confirmed 后还能原地修改。
8. 没有 confirmed 就生成 Case。
9. compiler 不支持还假装生成 Case。
10. LLM fake fallback 不提示。
11. 代码生成后不测试直接注册。
12. 把机理目标当成可编译参数。
13. 把热通量、Fr、对流边界这类歧义条件直接瞎填。
```

最终要让用户感觉：

```text
系统真的理解了我的复杂研究计划；
系统能先拆解，而不是乱套模板；
系统知道哪些条件已明确，哪些需要澄清；
系统知道哪些 case 当前能落地，哪些需要能力扩展；
系统给我的是科研实验草案，不是空参数表；
我可以通过自然语言修改；
每次修改前系统会说明影响；
我确认后系统才更新；
生成不了 Case 时，系统能明确告诉我缺什么。
```
