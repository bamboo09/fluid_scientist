# Fluid Scientist 完整重构方案

## 面向复杂、未知科研 Case 的开放式建模、能力解析、OpenFOAM 13 编译与验证系统

---

# 1. 项目目标

当前 Fluid Scientist 已具备以下基础能力：

* 用户研究需求输入；
* 科学意图解析；
* StudyDraft 生成；
* 多轮对话修改；
* CasePlan 生成；
* OpenFOAM 文件生成；
* 工作站连接；
* OpenFOAM 任务提交；
* OpenFOAMValidationRunner；
* ConfigExtensionExecutor；
* UnknownCapabilityOrchestrator；
* validation evidence；
* test manifest；
* VERIFIED 动态能力注册。

但当前系统存在两个核心问题。

## 1.1 OpenFOAM Case 生成质量不足

具体表现为：

* 模型盲猜 OpenFOAM 参数；
* OpenFOAM 版本混用；
* 字典语法错误；
* CasePlan 与编译结果不一致；
* patch 名称跨文件不一致；
* 湍流模型和字段不匹配；
* function object 参数不完整；
* 网格生成失败；
* 数值运行立即发散；
* 页面显示“生成成功”，提交后立即失败。

## 1.2 提高可靠性后可能退化成模板选择器

系统又不能采用：

```text
用户需求
→ 匹配完整案例模板
→ 没有模板
→ 无法处理
```

因为真实科研场景可能包括：

* 未见过的几何；
* 未见过的边界组合；
* 未见过的材料；
* 新的物理过程；
* 新的观测指标；
* 多区域、多物理场耦合；
* 多阶段工况；
* 复杂时变边界；
* 用户自定义分析需求。

完整案例模板永远不可能覆盖所有科研问题。

因此本轮系统需要同时实现：

> 已知能力可靠复用、已有能力动态组合、未知能力受控扩展、全流程真实验证。

---

# 2. 总体设计原则

本轮重构遵循以下原则。

## 2.1 LLM 负责理解，不负责保证语法

LLM 负责：

* 理解研究需求；
* 提取用户事实；
* 构建科学语义；
* 识别实体、区域和关系；
* 分解物理问题；
* 推断观测需求；
* 拆解原子能力；
* 发现未知能力；
* 生成 ExtensionSpec；
* 生成结构化 CaseIR Patch。

LLM 不负责：

* 直接生成完整 OpenFOAM 字典；
* 保证 OpenFOAM 语法；
* 直接修改最终 Case 文件；
* 注册未验证能力；
* 判定工作站运行成功。

---

## 2.2 Schema 负责忠实表达复杂问题

Schema 的目标不是提前枚举所有科学问题，而是能够表达：

* 实体；
* 区域；
* 几何关系；
* 区域接口；
* 物理过程；
* 材料；
* 初始条件；
* 边界条件意图；
* 多阶段工况；
* 观测目标；
* 参数来源；
* 假设；
* 歧义；
* 未解决要求；
* 能力缺口。

---

## 2.3 Capability System 负责判断和组合

系统不对整个 Case 简单分类成：

```text
圆柱
射流
管流
```

而是将 Case 拆解为原子能力需求，并逐项判断：

```text
已有
可组合
需扩展
需要新物理
需要澄清
环境受阻
```

---

## 2.4 Compiler 负责确定性生成

最终 OpenFOAM 文件必须由：

* 版本化 PlatformProfile；
* VERIFIED BasePack；
* VERIFIED Capability Components；
* 确定性映射规则；
* 确定性 Compiler；

共同生成。

相同输入必须产生一致输出。

---

## 2.5 ValidationRunner 负责证明可运行

系统不能用以下标准判定成功：

```text
生成了8个文件
JSON格式正确
controlDict里有solver
```

只有经过真实 OpenFOAM Foundation 13 环境验证，才能进入正式提交。

---

# 4. OpenFOAM 平台版本锁定

## 4.1 当前目标环境

当前工作站版本为：

```text
OpenFOAM Foundation 13
```

不是：

```text
OpenCFD v2406
OpenCFD v2312
OpenCFD v2506
```

必须清理所有 v2406 专属知识、配置和命令。

---

## 4.2 Foundation 13 基础运行方式

单相不可压缩流的基础求解模块：

```foam
solver incompressibleFluid;
```

运行命令：

```bash
foamRun -solver incompressibleFluid
```

主要文件：

```text
system/controlDict
system/fvSchemes
system/fvSolution

constant/physicalProperties
constant/momentumTransport

0/U
0/p
0/k
0/omega
0/nut
```

不得继续默认生成：

```text
constant/transportProperties
```

---

## 4.3 OpenFOAMPlatformProfile

新增统一平台配置：

```json
{
  "profile_id": "openfoam-foundation-13",
  "distribution": "OpenFOAMFoundation",
  "version": "13",
  "application": "foamRun",
  "default_solver_module": "incompressibleFluid",
  "physical_properties_file": "constant/physicalProperties",
  "momentum_transport_file": "constant/momentumTransport",
  "supports_dry_run": false,
  "security_policy": {
    "allow_code_stream": false,
    "allow_case_local_dynamic_libs": false,
    "allow_shell_variables": false,
    "allow_external_include": false,
    "allow_arbitrary_scripts": false,
    "allow_arbitrary_executables": false
  }
}
```

平台能力必须通过工作站探测确认：

```bash
foamVersion
foamRun -help
foamToC -solvers
foamToC -vectorBCs
foamToC -scalarBCs
foamToC -functionObjects
foamPostProcess -list
```

所有模块必须读取同一个 PlatformProfile。

原有 CasePlan 同时混合了科学意图、OpenFOAM 实现和运行配置，必须拆分。

新的核心数据模型：

```text
StudyDraft
RequestedCaseIR
AtomicRequirementSet
CapabilityResolutionPlan
ResolvedCaseIR
CompiledCaseManifest
ValidationManifest
```

# 6. StudyDraft

StudyDraft 用于面向用户展示和多轮确认，内容包括：

* 研究目标；
* 研究对象；
* 已知条件；
* 推荐参数；
* 待确认问题；
* 观测指标；
* 计算资源建议；
* 变更摘要。

StudyDraft 不是 Compiler 的直接输入。

同一 Session 中用户修改参数时：

```text
修改StudyDraft
→ 更新RequestedCaseIR相关路径
→ 生成变更摘要
→ 用户确认
→ 创建新CaseIR版本
```

不能每次都创建新的研究工作流。

# 7. RequestedCaseIR

## 7.1 定位

RequestedCaseIR 表达：

> 用户真正想研究什么。

它不要求系统已经具备实现能力，也不要求匹配完整模板。

---

## 7.2 Core Schema

建议结构：

```json
{
  "schema_version": "2.0",
  "case_ir_version": 1,
  "study_id": "study_xxx",
  "case_id": "case_xxx",

  "physics": {},
  "entities": [],
  "regions": [],
  "relations": [],
  "interfaces": [],
  "materials": [],
  "fields": [],
  "boundary_intents": [],
  "initial_conditions": [],
  "operating_stages": [],
  "mesh_intent": {},
  "numerical_intent": {},
  "observables": [],
  "derived_constraints": [],
  "assumptions": [],
  "ambiguities": [],
  "unresolved_requirements": [],
  "extensions": []
}
```

# 8. 参数统一结构

关键参数不能只保存一个 value。

统一使用：

```json
{
  "value": 3900,
  "unit": "dimensionless",
  "source": "USER_EXPLICIT",
  "confidence": 1.0,
  "status": "CONFIRMED",
  "assumption": null,
  "derived_from": [],
  "alternatives": []
}
```

支持的来源：

```text
USER_EXPLICIT
USER_CONFIRMED
MODEL_INFERRED
MODEL_RECOMMENDED
SYSTEM_DEFAULT
FORMULA_DERIVED
CAPABILITY_REQUIRED
TEMPLATE_DERIVED
LITERATURE_SUGGESTED
```

支持的状态：

```text
CONFIRMED
INFERRED
RECOMMENDED
ASSUMED
UNRESOLVED
AMBIGUOUS
CONFLICTING
```

# 9. 实体、区域、关系和接口

## 9.1 Entity

实体描述几何对象：

```json
{
  "id": "cylinder_1",
  "kind": "cylinder",
  "parameters": {
    "diameter": {
      "value": 0.01,
      "unit": "m"
    }
  }
}
```

---

## 9.2 Region

区域表示求解域：

```json
{
  "id": "fluid_region",
  "kind": "fluid",
  "material_ref": "air",
  "physics_refs": [
    "incompressible_flow"
  ]
}
```

复杂热耦合场景：

```json
{
  "id": "solid_wall_region",
  "kind": "solid",
  "material_ref": "steel",
  "physics_refs": [
    "solid_heat_conduction"
  ]
}
```

---

## 9.3 Relation

空间关系不能全部写死为 archetype。

例如：

```json
{
  "id": "relation_1",
  "type": "near",
  "source": "cylinder_1",
  "target": "wall_1",
  "parameters": {
    "gap": {
      "value": 0.2,
      "unit": "D",
      "reference": "cylinder_1.diameter"
    }
  }
}
```

可支持：

```text
near
inside
intersects
aligned_with
inclined_to
upstream_of
downstream_of
attached_to
rotates_about
moves_along
```

---

## 9.4 Interface

区域之间的耦合必须显式表达：

```json
{
  "id": "fluid_solid_interface",
  "region_a": "fluid_region",
  "region_b": "solid_region",
  "coupling_intent": "conjugate_heat_transfer"
}
```

不能只写：

```json
{
  "thermal": true
}
```

---

# 10. 单位与派生约束

所有物理参数必须带单位或明确为无量纲。

错误：

```json
{
  "diameter": 1
}
```

正确：

```json
{
  "diameter": {
    "value": 0.01,
    "unit": "m"
  }
}
```

无量纲关系必须保存：

```json
{
  "id": "reynolds_relation",
  "expression": "Re = U_ref * L_ref / nu",
  "inputs": [
    "reference.velocity",
    "reference.length",
    "physics.reynolds_number"
  ],
  "output": "materials.fluid.kinematic_viscosity"
}
```

新增：

```text
DimensionalConsistencyValidator
```

用于检查：

* 单位错误；
* 量纲不一致；
* Re、U、L、ν关系；
* 时间和频率关系；
* 热流与温度单位；
* 压力与速度尺度。

---

# 11. 多阶段工况

复杂 Case 可能包含：

* 初始静止；
* 入口渐增；
* 流动发展；
* 工况切换；
* 正式采样。

需要：

```json
{
  "operating_stages": [
    {
      "id": "initialization",
      "type": "initialization",
      "actions": [
        "set_quiescent_velocity"
      ]
    },
    {
      "id": "startup",
      "type": "transient_ramp",
      "time_range": [0, 2]
    },
    {
      "id": "development",
      "type": "flow_development",
      "time_range": [2, 20]
    },
    {
      "id": "sampling",
      "type": "measurement",
      "time_range": [20, 60],
      "observable_refs": [
        "drag_spectrum",
        "wake_flip"
      ]
    }
  ]
}
```

频谱分析不得使用启动阶段数据。

---

# 12. Observable Schema

观测指标不能只保存名称。

每个 Observable 至少包含：

* 科学目标；
* 目标对象；
* 需要的物理字段；
* 空间区域；
* 时间窗口；
* 采样频率；
* OpenFOAM 内部采样能力；
* 外部分析能力；
* 输出形式；
* 前置依赖；
* 当前能力状态。

示例：

```json
{
  "id": "wake_flip_metric",
  "semantic_type": "spanwise_wake_flip",
  "target_region": "wake_region",
  "required_fields": [
    "U"
  ],
  "sampling": {
    "type": "spanwise_probe_array",
    "frequency_requirement": "resolve_expected_shedding_frequency",
    "stage_ref": "sampling"
  },
  "analysis": {
    "method": "phase_and_sign_change_detection"
  },
  "capability_status": "UNRESOLVED"
}
```

必须区分：

### OpenFOAM 采样能力

```text
forces
forceCoeffs
probes
fieldAverage
wallShearStress
surface sampling
```

### 外部分析能力

```text
FFT
wake deflection fitting
vortex tracking
spanwise flip detection
horseshoe vortex detection
asymmetric separation analysis
```

---

# 13. 动态 Schema

不能为每个未知能力修改 Core Schema。

采用：

```text
Core Schema
+
Capability-specific Parameter Schema
```

例如：

```json
{
  "capability_id": "boundary.helical_pulsating_inlet",
  "parameter_schema": {
    "swirl_number": {
      "type": "number",
      "minimum": 0
    },
    "frequency": {
      "type": "number",
      "exclusiveMinimum": 0
    },
    "amplitude": {
      "type": "number"
    },
    "profile_type": {
      "enum": [
        "sinusoidal",
        "tabulated"
      ]
    }
  }
}
```

RequestedCaseIR 可先写：

```json
{
  "semantic_role": "helical_pulsating_inlet",
  "capability_ref": null,
  "parameters": {
    "swirl_number": 0.6,
    "frequency": 20,
    "amplitude": 0.15
  }
}
```

验证并注册新能力后，再绑定 capability_ref。

# 14. 大模型多阶段拆解 Pipeline

禁止采用：

```text
一次LLM调用
→ 直接输出最终CaseIR和OpenFOAM配置
```

必须使用多 Pass 流水线。

---

## Pass 1：Fact Extractor

只提取用户明确说出的事实。

输出：

* 实体；
* 参数；
* 初始条件；
* 边界；
* 时间顺序；
* 研究目标；
* 观测要求；
* 约束；
* 原始文本位置。

规则：

* 不补默认值；
* 不推荐 solver；
* 不映射 OpenFOAM；
* 不添加用户没有说过的事实。

---

## Pass 2：Ambiguity and Conflict Detector

识别：

* 同一参数多个值；
* 稳态与频谱冲突；
* 二维与展向翻转冲突；
* 等温与热流冲突；
* 周期边界和几何冲突；
* Re、U、D、ν不一致；
* 出口语义不明确；
* 研究目标互相矛盾。

输出：

```text
ambiguities
conflicts
blocking_unknowns
non_blocking_unknowns
```

---

## Pass 3：Scientific Normalizer

将用户描述标准化为科学语义。

例如：

```text
全场静止
→ quiescent initial velocity field

壁面黏住
→ no-slip wall intent

自然流出
→ unresolved outlet intent
```

如果存在多个候选解释，必须保留候选：

```json
{
  "raw_text": "对流出口",
  "candidate_concepts": [
    {
      "concept": "advective_outlet",
      "confidence": 0.62
    },
    {
      "concept": "non_reflecting_outlet",
      "confidence": 0.38
    }
  ],
  "status": "AMBIGUOUS"
}
```

---

## Pass 4：Entity / Region / Relation Builder

生成：

```text
EntityGraph
RegionGraph
RelationGraph
InterfaceGraph
```

识别：

* 哪些是几何实体；
* 哪些是流体区域；
* 哪些是固体区域；
* 实体之间的位置关系；
* 区域之间如何耦合；
* 哪些实体运动；
* 哪些接口发生传热或质量交换。

---

## Pass 5：Physics Decomposer

按照区域判断：

* 方程类型；
* 可压缩性；
* 相态；
* 热传递；
* 湍流；
* 材料模型；
* 外力；
* 旋转；
* 多孔介质；
* 运动和变形；
* 多区域耦合。

原则：

> 先判断需要求解哪些物理量和方程，再判断应该使用什么 OpenFOAM solver module。

---

## Pass 6：Observable Decomposer

将科学目标转化为：

```text
需要什么字段
在哪里采样
何时采样
多高采样频率
如何后处理
需要哪些前置能力
```

例如：

```text
分析尾迹翻转
→ 展向速度时序
→ 展向探针阵列
→ 稳定后的采样窗口
→ 足够采样频率
→ 翻转检测算法
```

不能直接将“尾迹翻转”映射为一个 probes function object。

---

## Pass 7：Atomic Requirement Decomposer

把需求拆成最小可实现能力。

近壁倾斜圆柱应拆成：

```text
单相不可压缩流
三维瞬态
LES WALE
圆柱几何
几何旋转
近壁定位
均匀速度入口
对流出口
无滑移边界
周期边界
尾迹网格加密
近壁网格加密
力系数采样
展向速度采样
尾迹偏斜分析
展向翻转分析
壁面涡结构分析
频谱分析
串行Smoke Test
并行Smoke Test
```

---

## Pass 8：Decomposition Critic

独立检查：

* 是否漏掉用户要求；
* 是否添加用户未说内容；
* 是否过早映射 OpenFOAM；
* 是否存在未拆开的复合需求；
* 是否遗漏多区域关系；
* 是否遗漏观测所需采样；
* 是否将新物理误判为普通配置；
* 是否提出了不必要澄清；
* 是否遗漏阻塞问题。

Proposer 和 Critic 可以使用同一个模型，但必须是独立调用和独立 Prompt。

---

# 15. Requirement Coverage

将用户原文拆成事实单元：

```text
F1 初始全场静止
F2 入口均匀来流
F3 圆柱无滑移
F4 壁面无滑移
F5 出口对流
F6 展向周期
F7 尾迹偏斜
F8 展向翻转
```

每个事实必须映射到以下之一：

```text
CaseIR path
Unresolved requirement
Clarification question
Explicit rejection
```

输出：

```json
{
  "coverage": 1.0,
  "facts": [
    {
      "fact_id": "F8",
      "mapped_to": "/observables/wake_flip"
    }
  ]
}
```

Coverage 未达到阈值时，禁止进入能力解析。

建议阈值：

```text
用户明确事实覆盖率 = 100%
```

# 16. 原子需求停止条件

某项需求满足以下条件时停止拆解：

1. 能映射到一个 Capability；
2. 或能映射到一个明确 ExtensionSpec；
3. 有清晰输入；
4. 有清晰输出；
5. 有独立验证方式；
6. 不同时跨越多个责任模块；
7. 进一步拆分不会改善编译和验证。

例如：

```text
periodic boundary pair
```

可以作为原子能力。

但：

```text
生成周期网格、运行LES并分析频谱
```

不能作为一个原子能力。

# 18. 澄清策略

只询问真正阻塞的问题。

## 18.1 必须询问

满足以下情况之一：

* 不回答就无法确定物理体系；
* 不同答案会导致不同 solver 或区域结构；
* 参数无法推导；
* 用户要求互相冲突；
* 存在重大安全或计算资源风险。

## 18.2 不需要询问

可通过以下方式解决：

* 公式推导；
* 合理系统默认；
* 模型推荐；
* 已验证能力默认值；
* Smoke Test 自动调整。

## 18.3 交互规则

* 一轮最多3个问题；
* 同类问题合并；
* 提供推荐选项；
* 解释不同答案影响；
* 用户回答后只更新相关 CaseIR 路径；
* 不重新生成整个研究草案。

* 依赖是否齐全；

# 21. Capability Resolution Engine

输入：

```text
AtomicRequirementSet
PlatformProfile
CapabilityRegistry
```

输出：

```text
CapabilityResolutionPlan
```

对每项能力判断：

## EXACT_SUPPORTED

已有完全匹配且经过验证的能力组合。

## COMPOSABLE_SUPPORTED

没有完整案例模板，但所有原子能力已存在，可以动态组合。

## EXTENDABLE

缺少局部能力，可以通过配置、代码或分析组件扩展。

## REQUIRES_NEW_PHYSICS

当前物理体系无法表达，需要新增 BasePack、solver module 或物理耦合能力。

## NEEDS_CLARIFICATION

缺少阻塞信息或存在科学矛盾。

## ENVIRONMENT_BLOCKED

工作站、编译器、MPI或安全环境无法满足。

# 22. 未知 Case 的分类方式

未知 Case 必须区分四种情况。

## 22.1 参数未见过

例如：

```text
Re=3900
→ Re=5000
```

直接修改参数并重新验证，不创建新能力。

## 22.2 组合未见过

例如系统已有：

* 倾斜圆柱；
* 近壁定位；
* 周期边界；
* LES；
* forceCoeffs。

但从未组合过。

处理：

```text
COMPOSABLE_SUPPORTED
→ 动态组合
→ 首次组合验证
→ 保存VerifiedComposition
```

## 22.3 原子能力未见过

例如新入口：

```text
脉动旋流入口
```

处理：

```text
EXTENDABLE
→ ExtensionSpec
→ 实现
→ 原子测试
→ 目标Case测试
→ 注册VERIFIED
```

## 22.4 新物理未见过

例如：

* 热流体；
* 多相；
* 共轭传热；
* 非牛顿；
* 多孔介质；
* 流固耦合。

处理：

```text
REQUIRES_NEW_PHYSICS
→ PhysicsExtensionSpec
→ 新BasePack或Solver能力
→ 基准验证
→ 目标Case验证
```

# 42. 最终验收标准

## 42.1 复杂需求理解

* 用户事实覆盖率达到100%；
* 用户未提及内容不会被当作事实；
* 歧义和冲突能够保留；
* 支持实体、区域、关系和接口；
* 支持多阶段工况；
* 支持参数来源、置信度和假设；
* 支持动态能力参数。

## 42.2 大模型拆解

* 使用多 Pass Pipeline；
* 不一次生成最终 Case；
* 具有独立 Critic；
* 能输出原子需求；
* 具有明确停止条件；
* 能构建依赖图；
* 只询问阻塞问题；
* 新物理不会误判为普通配置。

## 42.3 未知 Case

* 不依赖完整案例模板；
* 能识别参数变化、组合变化、原子能力变化和新物理；
* 能动态组合已有能力；
* 能受控扩展缺失能力；
* 扩展后能恢复原 Case；
* 未验证能力不能正式使用。

## 42.4 OpenFOAM 可靠性

* 唯一锁定 Foundation 13；
* 无 v2406 混用；
* 无 legacy solver 双重状态；
* LLM 不生成最终字典；
* Compiler 确定性；
* 跨文件验证完整；
* 网格验证完整；
* 串行和并行 Smoke Test；
* 未通过验证不能提交。

## 42.5 用户体验

* 同一 Session 内持续修改；
* 不重复询问已确认内容；
* 修改后仅更新相关路径；
* 页面展示真实处理阶段；
* 能查看能力缺口；
* 能查看失败原因；
* 不再出现“生成成功但提交立即失败”。

# 44. 核心结论

本系统最终不能是：

```text
自然语言
→ 模型生成OpenFOAM文件
→ 失败后继续让模型修文件
```

也不能是：

```text
自然语言
→ 匹配完整模板
→ 找不到模板就失败
```

最终架构必须是：

```text
LLM忠实理解复杂科研需求
→ Schema保存全部科学语义
→ 多阶段拆解原子能力
→ Capability Resolver判断已有、可组合、需扩展和新物理
→ 确定性Compiler生成OpenFOAM Foundation 13 Case
→ ValidationRunner在真实工作站证明可运行
→ 新能力验证后沉淀到Registry
```

核心原则：

> LLM 负责理解，Schema 负责表达，Capability System 负责判断与成长，Compiler 负责确定性生成，ValidationRunner 负责证明，只有真实验证通过的 Case 才能提交。
