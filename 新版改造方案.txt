# Fluid Scientist 主链智能化闭环改造实施方案

## 0. 任务定位

本轮不是重新设计项目，也不是推翻现有 Schema、模板编译器、三阶段确认交互和 OpenFOAM 远程执行体系。

本轮目标是：

> 在尽量少改动现有架构的前提下，把大模型、Skill、错误诊断修复、未知能力扩展真正接入当前 cylinder-flow 生产主链，使系统从“一次 LLM JSON 补全＋规则模板执行”升级为“可审计意图理解＋确定性编译＋真实执行反馈＋受控修复＋未知能力扩展”的闭环系统。

必须保留：

* `CylinderFlow2DExperimentSpecV1`
* `ObstacleFlowExperimentSpecV1`
* `SpecAdapter`
* `ObstacleFlowCompiler`
* 当前三阶段确认流程
* SSH/SCP 工作站执行
* OpenFOAM Foundation v13
* 当前前端总体布局和右侧面板
* 已实现的矩形、三角形、圆柱、底部轮廓编译能力

禁止通过以下方式完成任务：

* 删除 Schema，改成让模型直接输出完整 OpenFOAM Case；
* 让模型每次从零生成所有 OpenFOAM 文件；
* 新增更多静态模板掩盖意图理解问题；
* 使用 mock LLM 代替真实 GLM；
* 使用 mock OpenFOAM、固定日志或旧结果；
* 只新增类和接口但不接入生产路由；
* 只让单元测试通过，不运行真实端到端链路；
* 把 Skill 包装器执行成功当作 Skill 已生效；
* 仿真失败后重复运行同一份 Case；
* smoke test 失败后继续 full run；
* 静默使用 regex fallback；
* 将未知几何映射到“最接近”的已有几何；
* 为通过测试硬编码测试输入。

# 1. 当前问题基线

当前已确认：

1. GLM-4-Flash 仅在 `/api/v5/cylinder-flow/draft` 中调用一次；
2. GLM 只输出结构化 JSON，不生成当前主链中的 OpenFOAM 文件；
3. regex 抽取结果优先级高于 GLM，GLM 只能补空，无法纠正错误 regex；
4. 歧义检测、物理推导、模板选择、Case 生成、报告均是规则或确定性代码；
5. 4 个 Skill 只是函数调用包装器和审计记录器，不进入 Prompt，不修改 Spec，不改变 Compiler；
6. `ExtensionOrchestrator` 已有实现，但未接入生产主链；
7. OpenFOAM 执行失败后无模型诊断、无 Case 修改、无重试；
8. smoke test 失败仍会继续 full run；
9. Test D 中“正弦凸起”被同时识别为 rectangle 和 half_sine profile；
10. V5 核心状态使用内存字典，服务重启后流程丢失；
11. 当前运行代码存在大量未提交修改，不具备可靠复现基线；
12. 当前所有测试使用 mock LLM，真实模型路径缺少自动化测试。

本轮所有修改均围绕这些问题，不允许无关重构。

# 2. 总体目标架构

改造后的生产主链必须为：

```text
用户输入
  ↓
原始输入和会话持久化
  ↓
显式事实抽取器 Regex Extractor
  ↓
LLM Structured Interpreter
  ↓
Candidate Merger / Conflict Resolver
  ↓
Semantic Fidelity Guard
  ↓
Ambiguity & Requirement Critic
  ↓
用户澄清
  ↓
Canonical Experiment Spec
  ↓
Skill Resolver
  ↓
Capability Check
  ├─ 已支持
  │    ↓
  │  Deterministic Compiler
  │
  └─ 未支持
       ↓
     ExtensionOrchestrator
       ↓
     生成最小扩展
       ↓
     测试 + OpenFOAM v13 验证
       ↓
     注册 VERIFIED
       ↓
     恢复原任务
  ↓
用户确认实验方案
  ↓
编译
  ↓
静态验证
  ↓
网格验证
  ↓
Smoke Run
  ├─ 成功 → 等待用户确认正式运行
  └─ 失败 → Error Classifier → LLM Diagnosis → Controlled Repair → Retry
  ↓
Full Run
  ├─ 成功 → 后处理与物理验证
  └─ 失败 → Error Classifier → LLM Diagnosis → Controlled Repair → Retry
  ↓
结果分析
  ↓
LLM 科学报告 + 规则指标
  ↓
持久化展示
```

核心原则：

> LLM 负责理解、仲裁、诊断和提出修改；Schema 负责契约；Compiler 负责确定性生成；Validator 和真实 OpenFOAM 负责判定正确性。

# 4. 第二阶段：意图理解与候选仲裁

这是本轮最高优先级。

不要删除 regex，也不要让 GLM 完全覆盖 regex。

必须把当前“regex 优先、GLM 补空”改成“多候选提取＋字段级仲裁”。

## 4.1 保留两份独立结果

当前不能直接将 regex 和 LLM 修改到同一个 Spec。

新增：

```python
class ExtractionCandidate:
    field_path: str
    value: Any
    source: Literal["regex", "llm", "user", "formula", "default"]
    source_span: str | None
    confidence: float
    reasoning_summary: str | None
```

新增：

```python
class IntentCandidateSet:
    regex_candidates: list[ExtractionCandidate]
    llm_candidates: list[ExtractionCandidate]
    conflicts: list[CandidateConflict]
    unresolved: list[str]
```

流程：

```text
user_text
→ regex_extract_candidates()
→ llm_extract_candidates()
→ resolve_candidates()
→ canonical spec
```

禁止 regex 和 LLM 在候选阶段直接互相覆盖。

## 4.2 字段来源必须可追踪

所有关键字段必须保存：

```json
{
  "value": "triangle",
  "raw_value": "三角小障碍物",
  "source_span": "下壁面贴附一个高0.05m、宽0.1m的三角小障碍物",
  "source": "user_explicit",
  "regex_candidate": "triangle",
  "llm_candidate": "triangle",
  "resolution": "agreement",
  "confidence": 1.0,
  "confirmed": true
}
```

至少覆盖：

* domain length/height
* geometry type
* geometry dimensions
* geometry positions
* spatial relationships
* fluid
* Re
* inlet velocity
* viscosity
* boundaries
* initial conditions
* observables
* research goals

## 4.3 候选冲突分类

新增 `CandidateConflict`：

```python
class CandidateConflict:
    field_path: str
    regex_value: Any
    llm_value: Any
    raw_text: str
    conflict_type: Literal[
        "semantic_type_conflict",
        "duplicate_entity",
        "value_conflict",
        "spatial_conflict",
        "boundary_conflict",
        "unsupported_capability",
        "missing_required_field"
    ]
    severity: Literal["blocking", "warning"]
    resolution: str | None
```

仲裁规则：

### 情况 A：候选一致

```text
regex = triangle
llm = triangle
→ 直接接受
```

### 情况 B：regex 漏掉，LLM有明确来源

```text
regex = none
llm = triangle
source_span = “三角小障碍物”
→ 可接受，但经过语义忠实校验
```

### 情况 C：regex 和 LLM 冲突

```text
regex = rectangle
llm = half_sine_profile
```

不能固定 regex 胜出，也不能固定 LLM 胜出。

必须进入 `ConflictResolver`。

### 情况 D：同一句话被识别成两个实体

例如：

```text
正弦凸起，高5m、宽20m
→ rectangle
→ bottom_profile
```

应判断尺寸属于正弦凸起，而不是第二个矩形实体。

输出：

```json
{
  "conflict_type": "duplicate_entity",
  "resolution": "keep_bottom_profile_remove_rectangle"
}
```

### 情况 E：无法自动仲裁

生成用户澄清问题，不得静默选择。

## 4.4 LLM Prompt 改造

将现有 `_LLM_PARSE_SYSTEM_PROMPT` 拆成两个生产 Prompt。

### Prompt A：事实和实体提取

只允许提取用户明确表达的内容：

```text
不得补默认值。
不得计算派生参数。
不得将未知几何映射成近似几何。
不得将三角形映射为cosine_bell。
不得把正弦凸起同时创建为矩形实体。
必须为每个字段返回source_span。
无法确定时返回unknown。
```

输出严格 JSON Schema：

```json
{
  "entities": [],
  "boundaries": [],
  "physics": {},
  "observables": [],
  "spatial_relations": [],
  "unknown_terms": []
}
```

### Prompt B：冲突仲裁

输入：

* 用户原文；
* regex candidates；
* LLM candidates；
* 当前 Schema；
* 支持能力清单；
* 几何和边界冲突规则。

输出：

```json
{
  "resolved_fields": [],
  "blocking_conflicts": [],
  "clarification_questions": []
}
```

Prompt B 只能在存在冲突时调用，避免无意义增加成本。

## 4.5 模型调用参数

意图抽取使用真实 GLM，显式配置：

```text
temperature = 0
response_format = JSON
timeout
retry
model name
provider
```

不得使用默认随机参数。

开发环境和生产环境都必须显式指定 provider，禁止：

```python
LLMClient()
```

默认为 mock。

生产环境发现 provider=mock 时直接启动失败：

```text
PRODUCTION_LLM_PROVIDER_INVALID
```

## 4.6 LLM失败行为

当前静默 fallback 必须删除。

LLM失败后：

* 保存错误；
* 标记 `llm_status=FAILED`；
* 前端显示“模型理解服务暂时失败”；
* 允许用户选择“仅使用规则草案继续”；
* 不得用户无感知地自动降级。

只有用户明确选择规则模式，才继续。

# 5. 第三阶段：Semantic Fidelity Guard

新增生产模块：

```text
src/fluid_scientist/intent/semantic_fidelity_guard.py
```

必须在：

```text
候选仲裁完成后
Spec落库前
编译前
```

分别执行。

## 5.1 几何忠实性

建立显式同义词表：

```python
GEOMETRY_SYNONYMS = {
    "triangle": ["三角", "三角形", "三角障碍", "triangular"],
    "rectangle": ["矩形", "长方形", "rectangular"],
    "cosine_bell": ["余弦凸起", "余弦形凸起", "cosine bell"],
    "half_sine": ["正弦凸起", "半正弦凸起", "sinusoidal bump"]
}
```

规则：

* 用户明确写三角，不允许输出 cosine_bell；
* 用户明确写正弦凸起，不允许额外创建 rectangle；
* 用户明确写矩形，不允许输出 triangle；
* 未知几何不能映射到默认类型。

## 5.2 空间关系

支持并验证：

* attached_to
* below
* above
* centered_under
* centered_in_x
* distance_from_wall
* inside_domain
* non_intersection

例如：

```text
障碍物位于圆柱正下方
→ obstacle.center_x = cylinder.center_x
```

圆柱“正中央”和“距下壁2m”冲突时，不可静默解释。

## 5.3 几何相交检查

在编译前检查：

* 圆柱与壁面相交；
* 圆柱与障碍物相交；
* 障碍物超出流场；
* 三角形高度和宽度非正；
* 矩形和底部轮廓重复；
* 多个实体占据同一区域。

失败返回：

```text
SEMANTIC_GEOMETRY_INVALID
```

## 5.4 边界语义检查

验证：

* 左右入口出口是否合理；
* 周期边界是否成对；
* 2D front/back 是否一致；
* 上自由出流不会被错误生成成 no-slip；
* inlet/outlet patch 名称和 0/ 文件一致；
* 用户表达与最终 BoundaryConfig 一致。

# 7. 第五阶段：能力检查与未知能力接入

## 7.1 编译前能力检查

新增统一入口：

```python
capability_report = CapabilityResolver.resolve(canonical_spec)
```

输出：

```json
{
  "supported": [],
  "missing": [],
  "unverified": [],
  "required_extensions": []
}
```

不得再由单一 Compiler 默默决定是否支持。

## 7.2 未支持能力行为

遇到未知几何、边界、求解器或观测量：

```text
CAPABILITY_EXTENSION_REQUIRED
```

不得：

* 删除该字段；
* 使用默认值；
* 映射到近似类型；
* 跳过观测量；
* 继续编译错误Case。

## 7.3 接入 ExtensionOrchestrator

把现有 `extensions/orchestrator.py` 接入 cylinder-flow 主链。

生产路径：

```text
confirm-plan
→ capability check
→ missing capability
→ create extension checkpoint
→ ExtensionOrchestrator.execute(checkpoint)
→ validation
→ VERIFIED
→ resume original spec
```

## 7.4 Extension类型

优先顺序：

1. `CONFIG_EXTENSION`
2. `COMPILER_HOOK_EXTENSION`
3. `VALIDATOR_EXTENSION`
4. `POSTPROCESS_EXTENSION`
5. `CODE_EXTENSION`

能够用配置解决时禁止生成代码。

## 7.5 模型生成范围

未知能力代码生成只能生成最小 Patch。

输入必须包括：

* 当前仓库代码；
* 目标接口；
* 可修改文件白名单；
* OpenFOAM Foundation v13；
* 最近已验证实现；
* 必须通过的测试；
* 禁止修改项。

输出必须是：

```text
unified diff
```

禁止输出完整仓库或重写整个 Compiler。

## 7.6 扩展验证

必须执行：

```text
静态安全检查
Python导入检查
类型检查
单元测试
相关回归测试
生成Case
OpenFOAM字典检查
blockMesh/snappyHexMesh
checkMesh
smoke run
```

全部成功后才能注册：

```text
VERIFIED
```

失败时保存：

* patch；
* 测试日志；
* OpenFOAM日志；
* 模型调用；
* evidence manifest。

不能伪注册。

# 8. 第六阶段：OpenFOAM错误诊断与受控修复闭环

新增：

```text
src/fluid_scientist/repair/
```

至少包括：

```text
error_classifier.py
repair_context_builder.py
llm_diagnoser.py
repair_planner.py
controlled_repair_executor.py
repair_policy.py
```

## 8.1 错误分类

至少支持：

```text
FOAM_DICTIONARY_SYNTAX
MISSING_FILE
PATCH_MISMATCH
DIMENSION_MISMATCH
UNKNOWN_KEYWORD
OPENFOAM_VERSION_MISMATCH
BLOCKMESH_FAILURE
SNAPPY_FAILURE
CHECKMESH_FAILURE
CODED_BOUNDARY_COMPILE_FAILURE
SOLVER_STARTUP_FAILURE
NUMERICAL_DIVERGENCE
COURANT_TOO_HIGH
POSTPROCESS_MISSING_DATA
UNKNOWN
```

分类应先由规则完成，再让 LLM辅助解释。

## 8.2 修复上下文

不得把整个仓库和所有日志直接丢给模型。

根据错误类型收集：

```text
用户原始需求
Canonical Spec
OpenFOAM v13
当前阶段
相关文件
错误日志完整片段
前一轮修改
已通过阶段
允许修改的文件
```

例如 patch mismatch 只提供：

* `constant/polyMesh/boundary`
* `0/U`
* `0/p`
* `0/k`
* `0/omega`
* 当前错误日志

## 8.3 修复策略分级

### Level 1：确定性修复

不调用模型或模型只提供分类：

* patch名称同步；
* 缺失边界补齐；
* `forceCoeffs` patch修正；
* 文件缺失；
* known Foundation v13关键字修正；
* endTime/deltaT修正；
* Courant阈值修正；
* Cl列解析修复。

### Level 2：受控配置修复

模型输出：

```json
{
  "error_type": "...",
  "root_cause": "...",
  "repair_actions": [
    {
      "file": "system/controlDict",
      "operation": "replace_field",
      "path": "functions.forceCoeffs1.patches",
      "value": ["cylinder"]
    }
  ]
}
```

由程序执行，不允许模型直接写整个文件。

### Level 3：最小代码Patch

仅在配置无法修复时进入 ExtensionOrchestrator。

## 8.4 阶段冻结

已通过阶段默认不可修改。

例如：

```text
blockMesh PASSED
snappyHexMesh PASSED
checkMesh PASSED
solver startup FAILED
```

修复器默认不得修改几何和网格。

如需回退，模型必须明确说明原因，并由 RepairPolicy允许。

## 8.5 重试上限

每个阶段最多：

```text
3轮自动修复
```

防止无限循环。

每轮必须产生真实 diff。

若 diff 为空：

```text
RETRY_WITHOUT_REPAIR
```

禁止继续重试。

## 8.6 Smoke失败处理

立即修改当前逻辑：

```text
smoke FAILED
→ 禁止 full run
→ 进入错误诊断
```

只有 smoke 成功后才显示 Gate 3。

## 8.7 Full run失败

```text
full run FAILED
→ 收集日志
→ 错误分类
→ 修复
→ 重新从受影响阶段运行
```

不得只返回 FAILED。

# 9. 第七阶段：模型科学分析和报告

规则计算 Cd、Cl、St 应保留，因为它们是确定性指标。

新增模型分析层，但模型不能编造数据。

## 9.1 结构化结果摘要

先生成：

```json
{
  "job_id": "...",
  "mesh_quality": {},
  "time_range": {},
  "Cd": {},
  "Cl": {},
  "St": {},
  "force_signal_quality": {},
  "vortex_shedding": {},
  "warnings": [],
  "missing_data": []
}
```

## 9.2 LLM报告输入

模型只能读取：

* 用户研究目标；
* 已确认 Spec；
* 真实 OpenFOAM摘要；
* 真实指标；
* 图片和数据文件索引；
* 物理验证结果；
* 警告。

不得让模型自行猜测不存在的场。

## 9.3 报告输出

至少包含：

* 实验配置摘要；
* 网格和数值设置；
* 运行状态；
* Cd/Cl统计；
* 涡脱落频率与St；
* 尾迹形态分析；
* 初始过渡段说明；
* 数据质量和可信度；
* 未完成或缺失项；
* 是否足以支持研究目标。

## 9.4 物理验证

增加基础规则：

* Cd/Cl是否有限；
* 信号是否有足够样本；
* FFT是否有足够频率分辨率；
* St是否来自去除初始过渡后的Cl；
* 质量守恒或残差；
* 网格质量；
* Courant数；
* 是否达到周期稳定。

不通过时报告必须显示：

```text
RESULT_NOT_PHYSICALLY_VERIFIED
```

不能只因为求解器完成就写“实验成功”。

# 14. 强制端到端测试集

必须完成以下测试，全部通过后才能结束。

## Test A：标准圆柱绕流

输入：

```text
二维圆柱绕流，圆柱半径0.1m，来流速度1m/s，Re=200，分析Cd、Cl和涡脱落频率。
```

验证：

* 正确CylinderSpec；
* forceCoeffs配置；
* Cd、Cl、St；
* 用户三阶段确认；
* smoke通过；
* full run通过；
* 报告读取真实指标。

## Test B：圆柱＋矩形障碍

验证：

* rectangle仅一个；
* geometry diff；
* STL或网格变化；
* 结果目录独立。

## Test C：圆柱＋三角障碍

使用完整案例：

```text
在二维流场中，长10米，宽5米，圆柱半径0.1m，圆心距下壁面2m，位于流场正中央，来流速度1.0m/s，Re=200。下壁面贴附一个高0.05m、宽0.1m的三角小障碍物，位于圆柱正下方。左为速度入口，右为压力出口，上自由出流，下壁面无滑移。观测圆柱尾迹涡街形态、升阻力系数及涡脱落频率。
```

验证：

* triangle正确；
* cosine_bell禁用；
* source span正确；
* obstacle.center_x与cylinder一致；
* “正中央”和距下壁2m产生合理澄清；
* Compiler生成三角几何；
* checkMesh通过。

## Test D：正弦凸起

输入：

```text
二维长方形水流场，长300m、高25m，上边界施加向右切向应力，两侧周期，下壁无滑移；下壁中央有正弦凸起，高5m、宽20m；分析指定截面平均流速。
```

验证：

* half_sine profile启用；
* rectangle禁用；
* 无圆柱；
* 周期边界成对；
* 截面平均速度采集配置存在；
* 不进入圆柱专属逻辑。

## Test E：多轮修改

流程：

1. 创建圆柱；
2. 增加矩形；
3. 矩形改成三角形；
4. 增加Cd/Cl/St；
5. 修改入口速度；
6. 确认并编译。

验证：

* 同一个session；
* spec版本递增；
* 旧rectangle完全消失；
* 不新建无关联流程；
* 每轮有change summary。

## Test F：可控patch错误

故意制造：

```text
0/U patch name与网格不一致
```

验证：

* smoke失败；
* full run不启动；
* 错误分类为PATCH_MISMATCH；
* LLM收到相关日志和文件；
* 受控修复更新patch；
* 再次smoke成功；
* 有真实diff；
* 不是重复运行同一Case。

## Test G：数值发散

制造较大deltaT。

验证：

* 分类为COURANT_TOO_HIGH或NUMERICAL_DIVERGENCE；
* 修复策略修改deltaT；
* 不修改几何；
* 重新运行成功。

## Test I：重启恢复

# 20. 工作循环与停止规则

Trae必须按照以下循环工作：

```text
阅读真实代码
→ 定位生产调用
→ 做最小修改
→ 运行单元测试
→ 运行集成测试
→ 启动真实服务
→ 运行真实GLM请求
→ 运行真实OpenFOAM
→ 检查生成文件和结果
→ 发现问题
→ 定位根因
→ 修复
→ 回归测试
```

如果测试失败，不允许：

* 只改测试期望；
* 加硬编码；
* 降低断言；
* 使用mock替代；
* 跳过失败步骤；
* 声称“环境原因”后直接结束；
* 仅写TODO；
* 只提交设计文档。

必须继续定位，直到：

* 成功；
* 或获得明确且无法由代码解决的外部环境阻塞证据。

环境阻塞必须包含：

* 失败命令；
* 完整错误；
* 已尝试措施；
* 为什么不是代码问题；
* 恢复执行所需的最小外部条件。

不得在“代码已经写完但没有跑通”的状态下宣布完成。
